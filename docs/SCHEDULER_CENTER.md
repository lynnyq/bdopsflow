# BDopsFlow 调度中心功能说明文档

## 概述

BDopsFlow 调度中心是一个分布式、高可用的任务调度系统，支持定时任务、工作流编排和多执行器负载均衡。系统采用 Go 语言开发，使用 rqlite 分布式数据库和 Redis 缓存实现高可用架构。

## 核心功能

### 1. 任务管理

#### 1.1 任务类型
- **HTTP 任务**：发送 HTTP 请求，支持 GET/POST/PUT/DELETE 方法
- **Shell 任务**：执行 Shell 脚本
- **工作流任务**：作为工作流节点的子任务

#### 1.2 任务配置

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| name | string | 是 | 任务名称 |
| type | string | 是 | 任务类型：http、shell |
| config | object/string | 是 | 任务配置，支持对象或 JSON 字符串 |
| cron_expression | string | 否 | Cron 表达式，支持 5 位或 6 位（含秒）格式 |
| timeout_seconds | int32 | 否 | 任务超时时间（秒），默认 0（不限制） |
| retry_count | int32 | 否 | 最大重试次数，默认 3 |
| retry_interval | int32 | 否 | 重试间隔（秒），默认 5 |
| is_enabled | bool | 否 | 是否启用，默认 false |
| domain_id | int64 | 否 | 所属领域 ID，默认 1 |
| webhook_config | string | 否 | Webhook 回调配置 |
| assigned_executor_id | string | 否 | 指定执行器 ID |

#### 1.3 HTTP 任务配置
```json
{
  "url": "https://api.example.com/endpoint",
  "method": "GET",
  "headers": {
    "Authorization": "Bearer token",
    "Content-Type": "application/json"
  },
  "body": "{}",
  "timeout": 10000
}
```

#### 1.4 Shell 任务配置
```json
{
  "script": "echo 'Hello World'"
}
```

#### 1.5 任务生命周期
```
创建 → 启用 → 待调度 → 运行中 → 成功/失败 → 重试（失败时）
                    ↓
                  禁用 → 待启用
```

### 2. Cron 定时调度

#### 2.1 Cron 表达式支持
- **标准 5 位格式**：`分 时 日 月 周`
- **扩展 6 位格式**：`秒 分 时 日 月 周`
- 系统自动兼容两种格式

#### 2.2 调度器特性
- **分布式锁**：使用 Redis 实现任务锁，避免多实例重复执行
- **动态注册**：任务创建/更新时自动注册/更新调度
- **暂停/恢复**：支持全局暂停和恢复调度器
- **状态持久化**：调度器状态保存到 Redis，重启后自动恢复

#### 2.3 锁策略
- 锁超时时间 = 任务超时时间 × 2
- 最小锁超时：60 秒
- 最大锁超时：3600 秒
- 无超时限制的任务：锁超时为 600 秒

### 3. 执行器管理

#### 3.1 执行器注册
- 执行器启动时自动向调度中心注册
- 注册信息包括：ID、名称、地址、容量
- 支持重复注册更新信息

#### 3.2 心跳机制
- 执行器每 30 秒发送心跳
- 超过 60 秒无心跳自动标记为离线
- 离线执行器的任务自动失败

#### 3.3 负载均衡
- 选择策略：在线状态 + 当前负载 < 容量 + 随机
- 优先选择负载最低的在线执行器
- 支持指定执行器（assigned_executor_id）

#### 3.4 容量管理
- 动态调整执行器容量
- 容量变更通过 Redis 通知执行器
- 执行器下次心跳时应用新容量

### 4. 任务执行与重试

#### 4.1 执行流程
1. 调度器根据 Cron 触发或手动触发任务
2. 获取分布式锁，防止重复执行
3. 检查任务是否已在运行（数据库级检查）
4. 选择或使用指定执行器
5. 通过 gRPC 发送任务到执行器
6. 更新执行状态为 "running"
7. 等待执行结果
8. 更新执行状态为 "success" 或 "failed"
9. 触发 Webhook 回调（如果配置）
10. 根据结果决定是否重试

#### 4.2 重试机制
- 任务失败时自动重试
- 重试间隔可配置
- 达到最大重试次数后标记为失败
- 重试同样遵循负载均衡策略

#### 4.3 超时处理
- 执行器端超时：任务超时后执行器自动停止
- 调度器端超时：心跳检测超时后任务失败
- 两种超时都会触发重试

### 5. 工作流编排

#### 5.1 DAG 工作流
- 支持有向无环图（DAG）工作流
- 节点间可建立依赖关系
- 自动拓扑排序确定执行顺序

#### 5.2 工作流配置
```json
{
  "nodes": [
    {
      "id": "node1",
      "name": "任务1",
      "type": "http",
      "config": {
        "url": "https://api.example.com/1"
      },
      "position": { "x": 0, "y": 0 },
      "timeout_seconds": 60,
      "retry_count": 3
    }
  ],
  "connections": [
    { "from": "node1", "to": "node2" }
  ]
}
```

#### 5.3 工作流验证
- 检查节点存在性
- 检查重复连接
- 检查自循环
- 检查循环依赖

### 6. 分布式锁机制

#### 6.1 锁的类型
| 锁类型 | Key 格式 | 用途 |
|--------|----------|------|
| Cron 锁 | `cron:lock:task:{task_id}` | 防止多调度实例重复触发 |
| 执行锁 | `task:lock:{execution_id}` | 防止任务重复执行 |
| 续期锁 | `task:renew:{execution_id}` | 检测执行器存活 |

#### 6.2 锁续期机制
- 执行器心跳时自动续期相关任务锁
- 锁续期失败计数
- 连续失败 3 次强制失败任务

### 7. 故障处理

#### 7.1 卡住任务清理
- 每分钟检查一次
- 查找运行中但超过 5 分钟的任务
- 检查锁有效性
- 无效锁的任务强制失败

#### 7.2 离线执行器处理
- 自动标记超时未心跳的执行器为离线
- 离线执行器上的运行中任务失败
- 通知相关用户（通过 Webhook）

#### 7.3 任务重试
- 失败后等待重试间隔
- 选择新的执行器重试
- 记录重试次数

### 8. Webhook 回调

#### 8.1 回调配置
```json
{
  "url": "https://your-webhook.example.com/callback",
  "method": "POST",
  "headers": {
    "X-API-Key": "your-api-key"
  }
}
```

#### 8.2 回调事件
| 事件 | 触发时机 |
|------|----------|
| success | 任务成功完成 |
| failed | 任务失败（包括重试后失败） |
| skipped | 任务被跳过（上一次仍在运行） |

#### 8.3 回调 payload
```json
{
  "event": "success",
  "timestamp": 1234567890,
  "task_id": 1,
  "execution_id": "exec-1234567890-abc",
  "status": "success",
  "output": "任务输出",
  "error": "",
  "duration_ms": 1234,
  "metadata": {
    "task_name": "健康检查",
    "task_type": "http"
  }
}
```

### 9. RBAC 权限管理

#### 9.1 用户角色
| 角色 | 权限 |
|------|------|
| system_admin | 所有权限 |
| domain_admin | 所在领域的管理权限 |
| user | 基础权限 |

#### 9.2 领域隔离
- 任务、工作流、执行器都属于某个领域
- 用户只能看到和操作自己领域的资源
- 执行器可分配到多个领域

#### 9.3 权限模型
- 基于角色的权限控制
- 可配置角色与权限的映射
- 支持自定义权限

## 部署配置

### 环境变量
| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| DB_URL | rqlite 数据库地址 | http://localhost:4001 |
| REDIS_URL | Redis 地址 | redis://localhost:6379 |
| HTTP_PORT | HTTP 服务端口 | 8080 |
| GRPC_PORT | gRPC 服务端口 | 50051 |

### Docker 部署
```yaml
version: '3'
services:
  scheduler:
    image: bdopsflow/scheduler
    ports:
      - "8080:8080"
      - "50051:50051"
    depends_on:
      - rqlite
      - redis
  rqlite:
    image: rqlite/rqlite
    ports:
      - "4001:4001"
  redis:
    image: redis:alpine
    ports:
      - "6379:6379"
```

## 常见问题

### Q: 如何确保任务不重复执行？
A: 使用两层分布式锁机制：
1. Cron 触发时获取调度锁
2. 执行时获取执行锁

### Q: 执行器离线后任务如何处理？
A: 执行器离线后，运行中的任务会被自动标记为失败并触发重试（如果配置了重试）。

### Q: 工作流中的某个节点失败会怎样？
A: 默认情况下，工作流会在失败节点处停止，后续节点不会执行。

### Q: 如何实现任务的高可用？
A: 
1. 部署多个调度中心实例
2. 使用 rqlite 分布式数据库
3. 通过 Redis 分布式锁协调
4. 多个执行器做负载均衡和故障转移

### Q: 如何迁移旧的定时任务？
A: 使用 API 批量创建任务，支持从 JSON/YAML 导入任务配置。
