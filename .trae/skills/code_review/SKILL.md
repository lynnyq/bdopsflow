## 技能描述

本技能专为 BDopsFlow 分布式运维调度平台的代码审查设计，确保代码符合项目规范和最佳实践。

## 适用场景

- 代码提交前的审查
- Pull Request 审查
- 定期代码质量检查
- 安全漏洞审查

## 审查流程

### 1. 前置准备

- 阅读项目 [AGENTS.md](AGENTS.md) 文档
- 了解项目的目录结构和技术栈
- 获取需要审查的代码范围

### 2. 错误处理检查

**核心规则**：所有 error 必须接收校验，禁止使用 `_` 忽略错误

**检查要点**：
- [ ] 搜索代码中的 `_ =`，检查是否有忽略 error 的情况
- [ ] 确认每个 error 都有适当的处理
- [ ] 错误信息清晰有意义，包含上下文
- [ ] 必要时对错误进行包装

**示例**：
```go
// 正确
result, err := someFunction()
if err != nil {
    return fmt.Errorf("failed to do something: %w", err)
}

// 错误 - 禁止忽略
_ = someFunction()
```

### 3. 安全规范检查

**核心规则**：密码、地址、密钥统一从配置/环境变量读取，禁止硬编码

**检查要点**：
- [ ] 搜索代码中的硬编码密码、API key、密钥等
- [ ] 确认敏感信息从配置文件或环境变量读取
- [ ] 检查是否有在日志中输出敏感信息
- [ ] 确认数据库密码加密存储
- [ ] 检查 SQL 注入防护（使用参数化查询）

**Go 文件重点检查**：`scheduler/internal/datasource/driver/` 目录下的文件

**前端文件重点检查**：`web/src/api/` 目录下的文件

### 4. 日志规范检查

**核心规则**：统一项目日志组件，禁用 fmt.Println

**检查要点**：
- [ ] 搜索代码中的 `fmt.Println`、`fmt.Printf` 等
- [ ] 确认使用项目统一的日志组件
- [ ] 检查日志级别是否合理（debug/info/warn/error）
- [ ] 确认日志中不包含敏感信息
- [ ] 检查日志格式的一致性

**日志组件位置**：
- 调度中心：`scheduler/internal/logger/logger.go`
- 执行器：`executor/internal/logger/logger.go`

### 5. 依赖管理检查

**核心规则**：新增第三方依赖需要标注用途，不随意引入无用包

**检查要点**：
- [ ] 检查 `go.mod` 和 `web/package.json` 的新增依赖
- [ ] 确认每个新增依赖都有明确用途
- [ ] 检查是否有重复功能的依赖
- [ ] 确认依赖版本安全，无已知漏洞
- [ ] 检查是否可以用项目已有功能替代

### 6. 性能检查

**核心规则**：高频代码减少临时对象创建，兼顾内存与性能

**检查要点**：
- [ ] 检查循环内的对象创建
- [ ] 检查字符串拼接（优先使用 strings.Builder）
- [ ] 检查切片预分配（使用 make 时指定容量）
- [ ] 检查是否有不必要的深拷贝
- [ ] 检查并发安全问题

### 7. 代码修改原则检查

**检查要点**：
- [ ] 原有正常逻辑是否被不恰当地重构
- [ ] 是否遵循最小改动原则
- [ ] 新功能是否保持向下兼容
- [ ] 接口变更是否有版本控制
- [ ] 数据库迁移是否有回滚方案
- [ ] 大改动是否有修改思路备注

### 8. 代码质量检查

**检查要点**：
- [ ] 没有裸 panic（除非是初始化阶段的致命错误）
- [ ] 没有跨层直接操作 DB（遵循分层架构）
- [ ] 没有明显的冗余代码
- [ ] 没有过度设计和过度封装
- [ ] 代码可读性良好，有适当注释
- [ ] 函数和变量命名清晰
- [ ] 函数长度适中，职责单一

### 9. 测试检查

**检查要点**：
- [ ] 新增代码是否有相应的单元测试
- [ ] 测试覆盖关键路径和边界条件
- [ ] 测试是否可以独立运行
- [ ] 测试描述清晰

### 10. 文档检查

**检查要点**：
- [ ] 是否更新了相关文档
- [ ] 数据库变更是否更新了 schema.sql
- [ ] 配置变更是否更新了配置示例
- [ ] API 变更是否更新了文档

## 按文件类型的专项检查

### Go 文件专项检查

**文件位置**：`scheduler/` 和 `executor/` 目录下的 .go 文件

**检查清单**：
- [ ] 错误处理完整
- [ ] 使用 slog 或项目日志组件
- [ ] 没有硬编码敏感信息
- [ ] 遵循 Go 官方风格（使用 gofmt 格式化）
- [ ] 适当的注释
- [ ] 有相应的测试文件

### 前端文件专项检查

**文件位置**：`web/src/` 目录下的 .vue、.ts、.tsx 文件

**检查清单**：
- [ ] TypeScript 类型正确
- [ ] API 调用有错误处理
- [ ] 敏感信息不从前端硬编码
- [ ] 用户输入有验证和转义
- [ ] 组件设计合理
- [ ] 遵循 Vue 3 Composition API 最佳实践

### 数据库相关检查

**文件位置**：`deploy/schema.sql`、`deploy/migrations/` 目录

**检查清单**：
- [ ] 数据库变更有迁移脚本
- [ ] 迁移有回滚方案
- [ ] 数据完整性考虑周全
- [ ] 索引设计合理

## 审查报告格式

审查完成后，按以下格式输出报告：

```markdown
## 代码审查报告

### 总体评价
[总体评价：优秀/良好/需要改进/不通过]

### 发现的问题

#### 严重问题
- [ ] 问题描述
  - 位置：文件路径:行号
  - 建议：修复建议

#### 主要问题
- [ ] 问题描述
  - 位置：文件路径:行号
  - 建议：修复建议

#### 次要问题
- [ ] 问题描述
  - 位置：文件路径:行号
  - 建议：优化建议

### 改进建议
[具体的改进建议]

### 审查结论
[是否通过审查，需要哪些修改]
```

## 常用审查命令

```bash
# 运行所有测试
cd scheduler && go test ./internal/... -v
cd executor && go test ./internal/... -v

# 格式化检查
gofmt -l .

# 静态检查（如果安装了 golint/golangci-lint）
golangci-lint run

# 前端类型检查
cd web && npx vue-tsc --noEmit

# 搜索可能忽略的错误
grep -r "_ =" --include="*.go" .

# 搜索可能的硬编码
grep -r "password\|secret\|key\|api" --include="*.go" --include="*.ts" --include="*.vue" . | grep -v "config\|env\|test"
```
